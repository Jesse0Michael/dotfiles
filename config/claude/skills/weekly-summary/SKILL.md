Author: jesse0michael (jesse.michael1234@gmail.com)
Audience: yourself, your manager, standup notes. Length: 5–10 lines max.

## Purpose

Produce a tight summary of what **you** shipped this week. Not a changelog — a narrative of impact. If someone asks "what did you do last week?" this is the answer.

## Data collection

1. Fetch PRs authored by `jesse0michael` merged in the past 7 days.
2. Fetch commits by `jesse0michael` or `jesse.michael1234@gmail.com` on the default branch for the same window; dedupe against PRs.
3. Ignore: reverts, formatting-only changes, dependency bumps you didn't initiate, CI retriggers, regenerated protos / autogenerated client updates.
4. If something was added and then reverted within the same week, exclude it entirely — it nets to zero.

## Synthesis rules

- **Aim for 4–8 bullets**, but err toward more if clarity suffers. Over-clustering hides real work — a distinct effort deserves its own line.
- A bullet = **the same effort**, not adjacent topics. Different repo + different sub-problem usually = different theme, even when they share a domain noun ("activity," "auth"). Don't merge just because keywords overlap.
- **Read PR bodies, linked issues, and changed files when titles are ambiguous.** Many PRs titled by user-facing feature are actually tooling, codegen, or test-infra work — cluster by *real parent effort*, not literal title topic.
- **When a body of work centers on a specific tool or sub-system** (e.g. a test harness, codegen pipeline, internal service), make *that* the theme rather than the feature it ultimately serves. The shared scaffolding is usually the real story.
- **No umbrella catch-alls.** If the only honest header is "misc fixes" or "reliability fixes," the cluster is fake — split it. Name the kind of change precisely: *bug fix*, *new capability*, *hardening*, *test coverage*, *automation*, *migration*.
- Keep bullets short — sentence fragments are fine. Aim for under 12 words per bullet.
- When a theme spans multiple distinct sub-efforts, use sub-bullets. Keep sub-bullets to a short phrase.
- Lead with *what changed* for the user/system, not *what file you touched*.
- Skip mechanical follow-ups (lint fixes, review feedback iterations, typo corrections).
- If you reviewed others' PRs but didn't author code, don't include it unless the review drove a design decision worth noting.


## Output format

## Gopher Points

An uncapped points award based on vibes from four signals:

- **Impact** — how meaningful were the changes? Tiered judgment:
  - Cosmetic (logs, comments, renames) → minimal points
  - Hardening (edge cases, observability, perf, tests) → moderate points
  - New capability or significant behavior change → big points
- **Momentum** — did distinct effort actually land throughout the week, or was it one push and quiet days? Judge by *separate efforts shipped*, not PR count. Many small PRs on the same thing is one effort, not five.
- **Craft** — clean diffs, minimal fix-up commits after the fact.
- **Test coverage** — meaningful tests count for real points. Behavioral tests (not just "it constructs"), integration/e2e coverage on new code paths, or tests that lock in a tricky bug fix. Solid test work on a smaller feature can outscore a bigger untested one.

### Deflators (subtract)

- Heavy revision churn — many force-pushes or fix-up commits after merge
- Work that didn't really ship — merged behind a disabled flag, reverted later in the window, or still in draft
- PRs that are mostly logs, comments, or renames padding the count
- Scope-creep / kitchen-sink diffs that mix unrelated changes
- Mechanical follow-ups inflating the PR count (lint passes, retriggers, formatting)
- New code shipped without tests — especially new capabilities or non-trivial behavior changes

### Anchors

- Quiet maintenance week ≈ **8**
- Typical productive week ≈ **15–25**
- Exceptional week ≈ **35–45**

No formula. No cap. But don't reach for a big number — let the work justify it. The paragraph contextualizes why.
